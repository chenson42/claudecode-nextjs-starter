# Phase 1 — Functional Refinement (skipped per bug-fix variant)

**Bug:** A seeded admin signing in with valid credentials lands on `/access-pending` instead of `/admin`. The intended behavior is the seeded admin reaches the admin dashboard immediately (no 2FA challenge — the seeded admin is `twoFactorRequired: false`).

Confirmed real by reading the existing debug `console.log`s in `src/auth.ts` and the dev-server output: the JWT callback was correctly stamping `roles: ['admin']` + 4 features + `twoFactorRequired: false` into the token, yet every subsequent request was being redirected to `/access-pending` by `src/proxy.ts`.

# Phase 3 — Technical Design (skipped per bug-fix variant)

Root cause was unambiguous from the dev-server logs + `src/lib/auth/config.ts` having an empty `callbacks` block aside from `authorized`. Skipping a separate design phase. Fix sketch:

- Lift the `session` callback from `src/auth.ts` into the shared `authConfig` so the edge runtime (`edgeAuth` instance created in `config.ts`) projects custom claims onto `session.user`.
- The callback is a pure token→session transform with no DB access and no node-only imports — safe to share between Node and Edge runtimes.

## Implementer Notes

- Test was iterated to be **stable** rather than racy: an initial `waitForURL` would match transiently when the URL briefly hit `/admin` during the redirect chain before bouncing to `/access-pending`.
