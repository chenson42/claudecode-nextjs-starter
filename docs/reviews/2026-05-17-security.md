## HIGH

### H1 — Open redirect via unvalidated `callbackUrl` in TOTP verify action

**File:** `src/app/(auth)/totp/actions.ts` (lines 47, 85, 116)

`callbackUrl` is read from `formData` (user-controlled) and passed directly to Next.js `redirect()` without any same-origin check:

```typescript
const callbackUrl = String(formData.get("callbackUrl") ?? "/admin");
// ...
redirect(callbackUrl);  // called on success
```

An attacker who can submit the TOTP form with `callbackUrl=https://evil.com` can redirect a successfully-authenticated user off-site after they complete 2FA. The attack requires the user to complete a valid TOTP code (non-trivial), but the redirect then runs server-side before the browser sees it. Next.js 16 `redirect()` inside a server action does not enforce same-origin.

**Fix:** Strip the value to a relative path before use. A minimal guard:
```typescript
const raw = String(formData.get("callbackUrl") ?? "/admin");
const callbackUrl = raw.startsWith("/") && !raw.startsWith("//") ? raw : "/admin";
```
Apply the same guard to the signin page's `callbackUrl` usage (lower risk there since NextAuth's `signIn()` may validate `redirectTo`, but the pattern should be consistent).

## MEDIUM

### M1 — Email-change verification token stored in plaintext

**File:** `src/lib/db/schema.ts` (line 236), `src/app/(account)/account/actions.ts` (line 119), `src/app/(email-verify)/account/verify-email/[token]/page.tsx` (line 25)

The `emailVerificationTokens.token` column stores `crypto.randomBytes(32).toString("hex")` — the raw token is the DB value. A database read (via backup, logging, or a query-injection) exposes tokens that can be used to apply an arbitrary email change to any user who has a pending change. The `passwordResetTokens` table correctly hashes with SHA-256 before storage; this table should do the same.

**Fix:** Hash the token (SHA-256 hex) before inserting; look up by hash on consumption; delete on first use (already done). Follow the identical pattern used by `passwordResetTokens`.

### M2 — 2FA enrollment redirect loop for users with `twoFactorRequired=true` and no enrollment

**File:** `src/app/(auth)/totp/actions.ts` (line 52), `src/proxy.ts` (lines 44–51)

`verifyTotpAction` redirects unenrolled users to `/admin/2fa`. The proxy intercepts `/admin/2fa` as an admin route and, because `twoFactorRequired=true && !twoFactorVerified`, redirects back to `/totp`. The result is an infinite redirect loop that locks out any admin whose account has `twoFactorRequired=true` but has not yet enrolled.

The intended enrollment path is `/account/2fa` (a non-admin route, not gated by the 2FA proxy check). The fix in `verifyTotpAction` is to redirect to `/account/2fa` instead of `/admin/2fa` when no enrollment exists. Alternatively, add `/admin/2fa` to the proxy's 2FA-gate carve-out.

### M3 — X-Forwarded-For trusted without proxy verification (IP-based rate limits bypassable)

**File:** `src/lib/rate-limit.ts` (lines 48–57)

`getRequestIp()` trusts the first value in `x-forwarded-for` unconditionally. An attacker can set `X-Forwarded-For: 127.0.0.1` (or any IP) in their request to spoof the key used by IP-keyed rate limits (`signin`, `pwreset_req`). On Vercel this header is injected by the edge network and cannot be spoofed by the client, so the risk is production-deployment-specific. Local dev and non-Vercel deployments are vulnerable.

**Fix:** Document the assumption that the deployment platform is a trusted proxy that replaces (not appends) `x-forwarded-for`. Optionally, read only `x-real-ip` on Vercel, or accept the last value in a multi-hop `x-forwarded-for` chain.

### M4 — No server action for user deactivation; `isActive` can only be set by the seed script

**Files:** `src/app/(admin)/admin/users/page.tsx` (line 153), no corresponding `actions.ts`

The admin users page displays `isActive` but there is no server action to toggle it. An admin who needs to deactivate a user must update the database directly. More importantly, the `isActive=false` path in `src/auth.ts` (stale-JWT defense) works correctly — the problem is the missing administrative surface to set the value. This is a gap in the security tooling, not a bypass.

## LOW

### L1 — `changePassword` has no rate limit on failed attempts (self-documented gap)

**File:** `src/app/(account)/account/actions.ts` (line 245, annotated `NOTE (Gap 5)`)

An attacker with a stolen session token can brute-force the current-password check in `changePassword` without triggering any rate limit. `checkRateLimit` is called on other auth-adjacent actions but not here.

### L2 — Email address interpolated into HTML email body without escaping

**File:** `src/app/(account)/account/actions.ts` (lines 143–149)

The `newEmail` value is interpolated directly into the `html:` string template. The preceding regex (`/^[^\s@]+@[^\s@]+\.[^\s@]+$/`) permits `<` and `>` characters, so a value like `<img src=x>@domain.com` would pass validation and inject an HTML element into the confirmation email. Since the email is sent only to `newEmail` itself, the attacker is injecting into their own email — the impact is confined to phishing-style content in the confirmation message, not code execution in the app.

**Fix:** HTML-encode `newEmail` (replace `<`, `>`, `&` with their HTML entities) before interpolation, or use a templating layer that escapes by default.

### L3 — `assignRoleAction` / `removeRoleAction` accept unvalidated UUID strings from `formData`

**File:** `src/app/(admin)/admin/users/actions.ts` (lines 32–33)

`userId` and `roleId` are taken directly from `formData` without UUID format validation. SQL injection is not possible (Drizzle parameterizes), and a foreign key constraint catches non-existent IDs. However, a malformed or very long string could produce confusing error messages that leak internal schema details. Validate `userId` and `roleId` as UUIDs before the DB call.

## INFORMATIONAL

### I1 — Three moderate npm CVEs in `postcss` (transitive via `next` and `next-auth`)

`npm audit --omit=dev` reports 3 moderate vulnerabilities: PostCSS XSS via unescaped `</style>` in CSS Stringify output (`GHSA-qx2v-qp2m-jg93`), present in `next@^16.2.6` and `next-auth`. The suggested fix (`npm audit fix --force`) would downgrade Next.js to 9.x, which is not viable. Monitor the Next.js release channel for a patched version; the CVE is exploitable only when PostCSS output is written to HTML without sanitization, which is not how the app uses it.

### I2 — TOTP verify audit actions use raw string literals instead of `AUDIT_ACTIONS` constants

**File:** `src/app/(auth)/totp/actions.ts` (lines 76, 83, 91, 105, 112)

`"totp.verify_failed"`, `"totp.verify_succeeded"`, `"totp.recovery_failed"`, `"totp.recovery_succeeded"` are inlined rather than added to `AUDIT_ACTIONS` in `src/lib/audit.ts`. No security impact, but a string rename would silently break historical audit queries. Add the four values to the constants catalog.
