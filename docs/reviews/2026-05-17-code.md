## Critical

### C-1: TOTP audit strings bypass the AUDIT_ACTIONS catalog
**File:** `src/app/(auth)/totp/actions.ts:76,83,91,105,112`

`verifyTotpAction` writes four audit strings as raw inline literals (`"totp.verify_failed"`, `"totp.verify_succeeded"`, `"totp.recovery_failed"`, `"totp.recovery_succeeded"`) rather than via the `AUDIT_ACTIONS` catalog in `src/lib/audit.ts`. These strings are not in the catalog at all. Consequences: (1) no static constant to search for; (2) strings can drift silently from any future analytics or alerting that keys on audit action names; (3) violates the pattern every other action file follows. Add four entries to `AUDIT_ACTIONS` and replace the four inline literals.

### C-2: `isFlagEnabled` is defined but never called anywhere in the app
**File:** `src/lib/flags.ts`

`isFlagEnabled` exists and is correct in isolation, but zero pages, actions, or route handlers import it. The `demo.new_dashboard` flag seeded by `scripts/seed.ts` also goes unchecked. The flags page lets admins toggle flags that have no effect. This is either dead infrastructure or a missing integration. For a starter that explicitly teaches the flags pattern, the absence of a single real call site is a gap that forks will blindly copy.

### C-3: `/account` subtree routes are not listed in `PUBLIC_PATHS` but are also not explicitly protected
**File:** `src/proxy.ts:5-14`

The proxy's `PUBLIC_PATHS` set covers `/`, `/signin`, `/totp`, `/access-pending`, `/forgot-password`, `/reset-password`. The `(account)` route group (`/account`, `/account/2fa`) does not appear in `PUBLIC_PATHS` and does not match the `PROTECTION_RULES` array. The fallthrough at line 64 allows any authenticated user through â€” which is the correct behavior for the account shell. However this is implicit rather than explicit: a reader extending `PROTECTION_RULES` or `PUBLIC_PATHS` cannot tell whether `/account` was intentionally omitted. An inline comment at that fallthrough explaining "account/* is authenticated-only, no role required â€” intentional fallthrough" would eliminate the ambiguity. As written it is not a runtime bug but it is a quiet invariant violation: the proxy's contract is undocumented for this path family.

## Notable

### N-1: Duplicate `setFreshRecoveryCodesCookie` and `replaceRecoveryCodes` helpers
**Files:** `src/app/(admin)/admin/2fa/actions.ts:26-48`, `src/app/(account)/account/2fa/actions.ts:30-53`

Both files define identical `setFreshRecoveryCodesCookie` and `replaceRecoveryCodes` helpers, differing only in the cookie `path` attribute (`/admin/2fa` vs `/account/2fa`). The business logic (generate codes, delete old codes, insert hashed codes, set cookie) is copy-pasted. Any bug fix in one will not propagate to the other. Extract to `src/lib/two-factor.ts` (which already owns TOTP primitives) or a shared `src/lib/recovery-codes.ts`, parameterizing the cookie path.

### N-2: `flags.ts` â€” `isFlagEnabled` hits the DB on every call with no caching
**File:** `src/lib/flags.ts:5-10`

Each call issues a `findFirst` query with no caching, memoization, or TTL. For a flag that is checked on every page render (once the C-2 gap is fixed), this is a per-request DB hit for data that almost never changes. Acceptable for the starter baseline, but worth a TODO comment so forks that add flag checks to hot paths know to add a cache layer.

### N-3: `(admin)/admin/2fa/page.tsx` generates a new TOTP secret on every Server Component render
**File:** `src/app/(admin)/admin/2fa/page.tsx:131-149`

When a user has no enrolled TOTP, the page calls `encryptSecret(generateSecret())` and upserts a `userTotpPendingEnrollments` row on every request. A user who refreshes the enrollment page repeatedly before completing it will silently invalidate their previous QR code. The `(account)/account/2fa/page.tsx` version handles this correctly â€” it checks for an unexpired pending row before minting a new one (lines 97-118). The admin path should do the same. This is a UX defect (the user may scan a QR code that has already been replaced) and a minor DB churn issue.

### N-4: `src/components/` does not have `admin/` or `shared/` subdirectories
**Files:** `src/components/` (directory listing)

CLAUDE.md â†’ Directory Structure documents `src/components/admin/` and `src/components/shared/`. Neither directory exists. Admin-specific components (`two-factor-card.tsx`) are co-located with their page under `src/app/(admin)/admin/users/[id]/`. Account components (`profile-form.tsx`, `email-form.tsx`, etc.) live under `src/app/(account)/account/`. This is workable for the current feature set but diverges from the documented shape, which downstream forks expect to find. Either create the directories and move components, or update CLAUDE.md to document the actual co-location pattern.

### N-5: `src/app/(admin)/admin/users/page.tsx` has no page-level permission check
**File:** `src/app/(admin)/admin/users/page.tsx:1-208`

The page reads and renders user data with no call to `auth()` or `hasFeature()`. The proxy enforces the `admin.users` feature check at the URL level (line 17 of `proxy.ts`), and the layout enforces authentication. But the page-level gap means: (a) if a future refactor moves the route outside the proxy's pattern, the page will silently expose user data; (b) it breaks the stated rule in CLAUDE.md that "page-level checks enforce per-feature access." The sibling `[id]/page.tsx` does this correctly. Add an `auth()` + `hasFeature(FEATURES.ADMIN_USERS)` guard to the page.

### N-6: `changePassword` in account actions lacks rate limiting
**File:** `src/app/(account)/account/actions.ts:245-251` (comment at line 245)

The code itself notes: "No rate limiting on failed password-change attempts. An attacker with a stolen session token could brute-force the current password." This is accurate. Unlike the Credentials sign-in, TOTP verify, password reset request, and password reset consume paths â€” all of which call `checkRateLimit` â€” the `changePassword` action has no limit. Given the starter already has `checkRateLimit` as a zero-config primitive, this is a straightforward fix.

## Minor

### M-1: `console.warn` calls in production path of `src/lib/email.ts`
**File:** `src/lib/email.ts:25-28`

The two `console.warn` calls are intentionally gated behind `NODE_ENV !== "production"` â€” the production path throws instead. The invariant ("no `console.log` in production paths") is technically respected. However, `console.warn` in a dev-only guard is borderline; a dedicated logger or a no-op would be cleaner. Low priority.

### M-2: `requestAccountDeletion` is a stub with no user-facing disclosure
**File:** `src/app/(account)/account/actions.ts:258-283`

The action returns `{ ok: true }` and shows a toast saying deletion is not implemented. The `delete-button.tsx` UI uses the destructive `AlertDialog` pattern correctly. The stub is intentional and documented. The only minor issue is that the button is fully visible and interactive in the UI â€” a user completing the flow will be surprised by the toast. A disabled state or a note in the UI that the feature is coming would be less surprising.

### M-3: `src/app/(account)/account/2fa/page.tsx` uses a dynamic import inside a Server Component
**File:** `src/app/(account)/account/2fa/page.tsx:109`

Line 109 uses `await import("@/lib/two-factor")` to get `decryptSecret` and `otpauthUrl` when re-using an existing pending enrollment. This works correctly but is unusual: `two-factor.ts` is a Node-only module (uses `node:crypto`) that could be statically imported at the top of the file. The dynamic import does not provide any benefit here (no code-splitting gain in a Server Component). It should be a static top-level import.

### M-4: `PENDING_TTL_MINUTES` is defined in three places
**Files:** `src/app/(admin)/admin/2fa/page.tsx:42`, `src/app/(account)/account/2fa/page.tsx:16`, `src/app/(account)/account/2fa/actions.ts:27`

The constant `10` (minutes for pending TOTP enrollment expiry) is defined independently in each file. If the TTL changes, all three must be updated. Export it from `src/lib/two-factor.ts` as `TOTP_PENDING_TTL_MINUTES`.

## Observations

**O-1: Edge-runtime constraint is well-respected.**
`src/proxy.ts` and `src/lib/auth/config.ts` import only `next-auth`, `next/server`, and the internal `permissions.ts` constant. No DB, no Node-only modules. The separation is clean.

**O-2: Permissions vs flags separation is correct.**
`FEATURES` + `hasFeature()` is used for per-user authorization; `isFlagEnabled` (though uncalled â€” see C-2) is clearly kept separate. The schema comment at `src/lib/db/schema.ts:177-179` explicitly documents the distinction. No conflation found.

**O-3: Audit coverage on security mutations is thorough.**
Every security-sensitive mutation â€” role assignment, role removal, 2FA enrollment, recovery code regeneration, 2FA reset, flag toggle, email change, password change, password reset â€” writes to `audit_events`. The only gap is the inline TOTP verify/fail strings noted in C-1.

**O-4: Server/client split is consistently correct.**
Every `'use client'` component has a legitimate reason (event handlers, hooks, `useState`). No `'use client'` was found in layout files or pages that do not need interactivity. No DB access was found in client components.

**O-5: `src/auth.ts` is the largest single file at 233 lines.**
It is not unwieldy â€” the length is driven by the JWT callback's two DB queries and the documented role/feature loading logic. The logic is well-commented. Not a hotspot.

**O-6: `isFlagEnabled` ignores `rolloutPercent`.**
`src/lib/flags.ts` reads `enabled` only. The `rollout_percent` column exists in the schema and is seeded but is never evaluated. For a starter this is fine and documented by the schema comment. Forks that want percent-based rollouts will need to add the hashing logic.
