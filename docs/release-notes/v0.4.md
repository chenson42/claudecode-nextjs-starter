## 0.4.0 — 2026-05-18

### Feature: Upstream-Sync Periodic Review

**Value:** The starter is *meant* to be forked, but until now there was no mechanism to ask "is there anything new in the canonical starter I should pull into my fork?" Forks drift silently — bug fixes (like v0.3.4's timezone fix), security patches, dependency bumps, and agent refinements landed upstream but never made it back to long-lived forks. This release adds a periodic-review skill that closes that loop without forcing the fork-owner to actually merge anything: it produces a *punch-list*, the fork-owner decides what to cherry-pick.

#### What's New

- **A new periodic-review type — `upstream-sync` — bringing the total from seven to eight.** The cadence-check at session start now also flags overdue upstream syncs (every 14 days while a fork is active).
- **A new skill at `.claude/skills/upstream-sync/SKILL.md`.** Invoked manually (`/upstream-sync`) or when the cadence-check surfaces it as overdue. The skill:
  - Pre-flight: detects whether it's running inside the canonical starter (via `git remote get-url origin` compared against the hardcoded canonical URL) and exits cleanly if so — the review is fork-only.
  - Reads `.claude/upstream-state.json` for the fork's `forkPointSha`, `lastSyncedSha`, and `personalizedPaths`. First-run path derives `forkPointSha` from `git merge-base` or prompts the user.
  - Pulls upstream commits since `lastSyncedSha` via `gh api repos/...` (preferred — no remote needed) with a `git fetch <upstream>` fallback. Both failure modes exit cleanly without writing to the review log.
  - Classifies each commit: **must pull** (security, defect fix), **should pull** (general bug fix), **optional** (feature you may or may not want), **skip** (release-housekeeping noise). Heuristic is published in the skill — fork-owners can override by inspection.
  - Outputs a markdown table with SHA, date, author, subject, classification, files-touched (so a structurally-diverged fork can assess relevance), and a conflict flag (set when any touched file appears in `personalizedPaths`).
  - On a non-empty run, writes `docs/reviews/YYYY-MM-DD-upstream-sync.md` and appends to `docs/reviews/log.md`. On a failed run, writes **nothing** so the cadence warning keeps firing.
- **`personalize-starter` now initializes `.claude/upstream-state.json`.** New Step 8 in that skill records the canonical upstream URL, the fork-point SHA, and the list of files the personalize pass rewrote (so the upstream-sync skill knows where conflicts are likely).
- **`tech-lead` agent now owns the upstream-sync review** alongside the retrospective (7d) and documentation review (30d). New ownership bullet added to `.claude/agents/tech-lead.md`.

### Defect Fix (bundled): ESLint False Positive in v0.3.4 e2e Spec

**Problem:** The Playwright spec added in v0.3.4 (`e2e/timezone-safe-dates.spec.ts`) used `new Date(iso).toLocaleString()` inside `page.evaluate(...)` to independently compute the expected locale-formatted string for each browser context. The v0.3.4 ESLint rule that bans `toLocaleString` outside the `<FormattedDate>` primitive caught this call as a violation. `npm run lint` had been failing on `main` since the v0.3.4 commit.

**Root Cause:** ESLint operates on source text; it cannot distinguish a `toLocaleString` call that runs server-side (the bug the rule guards against) from one that runs inside a Playwright `page.evaluate` callback (which executes in the browser).

**Fix:** Inline `eslint-disable-next-line no-restricted-syntax` on the offending line with a comment explaining the page-evaluate context. The narrower disable is preferred over exempting all of `e2e/**` because the rule's intent (catch accidental TZ leaks) still applies to real e2e code.
