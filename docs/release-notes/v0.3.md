## 0.3.4 — 2026-05-18

### Defect Fix: Timezone-Safe Date Rendering

**Problem:** Every timestamp in the admin and account surfaces (last-login, account-created, TOTP enrolled-on, TOTP last-used) rendered in the *server's* timezone — UTC on Vercel — regardless of the viewer's actual timezone. The client component on the user-detail page additionally produced a React hydration mismatch whenever the client's local TZ differed from UTC.

**Root Cause:** Seven call sites across five files used `Date.prototype.toLocaleString()` / `.toLocaleDateString()` directly. In a Server Component that formatting runs on the server, so the rendered text is bound to the server's TZ. In a Client Component the SSR pass uses the server's TZ while the hydration pass uses the client's, causing a mismatch React logs to the console.

**Fix:** A single client primitive at `src/components/shared/formatted-date.tsx` that renders `<time dateTime={iso}>` with the ISO date slice (`YYYY-MM-DD`) as the SSR fallback and `suppressHydrationWarning`, then swaps in `toLocaleString()` / `toLocaleDateString()` after mount via `useEffect`. The primitive accepts `Date | string | number`, normalizes internally, and guards against invalid input (renders an em-dash). All seven call sites now use `<FormattedDate value={...} mode="date|datetime" />`.

To prevent regrowth, an ESLint `no-restricted-syntax` rule in `eslint.config.mjs` bans `toLocale{String,DateString,TimeString}` everywhere except the primitive's own file. A new Key Invariant in `CLAUDE.md` ("Timezone-Safe Date Rendering") and a one-liner in `.claude/agents/ux-developer.md` close the documentation loop.

**Testing:**
- [x] `npm run typecheck` — pass
- [x] `npm run lint` — pass; rule smoke-tested by temporarily inserting a `toLocaleString()` call elsewhere and confirming the rule fires
- [x] `npm run test` — 154 / 154 pass
- [x] `npm run test:e2e` — new spec passes in 4.3s against a fresh dev server with `AUTH_URL` matched to the dev port; two-TZ assertion confirms the fix end-to-end
- [x] `npm run check:audit` — zero violations

**Decision logged:** DECISION-007 in `docs/decisions.md` — primitive placement under `src/components/shared/`, ESLint approach (`no-restricted-syntax` with path-scoped exemption), and the SSR-fallback choice (ISO date slice + `suppressHydrationWarning`).

**Work-log:** `docs/work-log/2026-05-18-timezone-safe-dates.md`.

**Known follow-ups** (each gets its own work-log next session):

1. **Node engine floor** — `.nvmrc` says `20` and `package.json` `engines` requires `>=20.9.0`, but a developer who lands on Node 18 hits cryptic Vitest 4.x / Next 16 failures rather than a clear error. Owner: `deployment-engineer`.
2. **e2e port-collision footgun** — when a sibling project holds port 3000, the starter's dev falls back to 3001 but `AUTH_URL` in `.env.local` still points to 3000, so NextAuth's post-signin redirect crosses to the wrong app. Document in the e2e README or `pre-push` skill. Owner: `tech-lead` or `deployment-engineer`.
3. **ESLint blind spots** — the new rule catches `toLocale*()` method calls but not `new Intl.DateTimeFormat(...)`, `.toString()`, or manual template-literal assembly. Low priority; the right resolution is probably a `docs/decisions.md` entry naming the gap and a conscious choice about whether to extend the rule. Owner: `tech-lead`.

## 0.3.3 — 2026-05-18

The "Known follow-ups" punch-list from v0.3.2 — all seven items addressed in one sweep. The starter exits v0.3 with zero deferred review findings.

### Test coverage on the two zero-coverage critical modules

`src/lib/flags.ts` and `src/lib/two-factor.ts` were flagged at 0% coverage across three consecutive QA reports (v0.2 test-coverage review, every Phase 5 since, and v0.3.2's retrospective). Both now have unit-test suites:

- `src/lib/flags.test.ts` — 6 tests, 100% statement coverage. Covers unknown key, enabled/disabled, null vs undefined row, DB-call assertion, error propagation. One real finding: `isFlagEnabled` ignores the `rollout` percent column on `feature_flags`. Documented as a v0.4 candidate; doesn't block this release.
- `src/lib/two-factor.test.ts` — 28 tests, 100% statement coverage. Covers AES-GCM encrypt/decrypt roundtrip, unique-IV-per-call, tampered-ciphertext rejection, missing/wrong-length key rejection, `generateSecret` randomness + base32 format, `verifyToken` accepts current code + rejects expired + rejects wrong, `otpauthUrl` format, `generateRecoveryCodes`, `hashRecoveryCode`, and every branch of `normalizeRecoveryCode`.

### Bug fix: admin `/admin/2fa` minted a new TOTP secret on every render

The admin-side enrollment page (`src/app/(admin)/admin/2fa/page.tsx`) called `generateSecret() → encryptSecret() → upsert pending-row` on **every render** when the user had no enrollment. Each request invalidated any QR the user had already scanned in another tab — their authenticator app's code would never verify against the new secret. The user-side `/account/2fa` page had been fixed earlier to check for an unexpired pending row first.

**Fix shape:** extracted the "get-or-create pending enrollment" logic into a shared helper at `src/lib/totp-pending.ts`. Both `/admin/2fa` and `/account/2fa` now call it. DRY win as a side effect — the inline check-then-reuse block plus a `dynamic import("@/lib/two-factor")` in the account page are gone. 9 new unit tests on the helper.

### Security hardening: TOCTOU on `consumeResetToken`

Carry-forward from v0.3.0 Phase 6. The original implementation read the token row outside any transaction, then deleted-and-updated inside one — a TOCTOU window where two concurrent requests could both observe the token as valid before either consumed it.

**Fix:** rewrote `consumeResetToken` to use a single `DELETE ... WHERE token=? AND expiresAt>now() RETURNING *` statement. Postgres guarantees the row is claimed by exactly one statement; the second concurrent request gets `[]` and aborts with the same friendly error. The `db.transaction()` wrapper is gone (no longer needed once the token is atomically claimed before any subsequent writes). 3 new regression tests simulate the double-submit race.

### Dependency bump: `@neondatabase/serverless` 0.10.4 → 1.1.0

Major version. v1.x's three breaking changes (template-tag-only `sql()` invocation, Node <19 dropped, inlined type declarations) don't touch how the starter consumes the driver — `neon(url) → drizzle(sql, {schema})`. Zero application code adapted. Verified against the dev DB; `npm run build` + e2e all green.

`drizzle-orm` `^0.45.2` is fully compatible with Neon 1.x — no compounding bump needed.

### Agent + README polish

- **`.claude/agents/api-developer.md`** — description trimmed. "Schema changes" removed from the responsibilities list (that's `database-admin`'s ownership). Reduces routing-time boundary confusion the agent-instruction review flagged.
- **`.claude/agents/deployment-engineer.md`** — env-var table expanded. Was missing 6 entries (`NEXT_PUBLIC_APP_URL`, `INITIAL_ADMIN_EMAILS`, `SEED_ADMIN_*`, `RESEND_*`, `UPSTASH_REDIS_*`, `TRUST_PROXY_HEADERS`, `RATE_LIMIT_DISABLED`). Now organized as Required / Required-for-Credentials / Required-for-email / Optional.
- **`README.md` forking notes** — added a "Run `/personalize-starter` first" callout. Also added a "Heads-up on dependencies" section that names the NextAuth-beta tradeoff explicitly. The dependencies review flagged this as a known risk for production forkers; calling it out in the README means they don't discover it by surprise on launch day.

### Security fixes (1 HIGH + 4 MEDIUM, all from this PR's review)

- **HIGH — Open redirect via `callbackUrl` in TOTP verify** (`src/app/(auth)/totp/actions.ts`). Added `sanitizeCallbackUrl()` that rejects any value not matching `/^\/(?!\/)/`. ### Fix: lint was silently failing in CI

The `npm run lint` script invoked ESLint without a config file. ESLint 9 dropped support for `.eslintrc.*` as the default, so it errored out immediately. The CI workflow had `continue-on-error: true` on the lint step, which masked the failure as success — meaning the lint gate had been a no-op since the repo was created.

**What changed:**

- Added flat-config `eslint.config.mjs` that consumes `eslint-config-next`'s bundled flat-config array (covers next, react, react-hooks, typescript, jsx-a11y, import). Ignores `.next/`, `out/`, `build/`, `coverage/`, `test-results/`, `playwright-report/`, `drizzle/`, `next-env.d.ts`.
- Removed `continue-on-error: true` from the CI Lint step — failures now actually block the build.
- Strict-warning mode: `npm run lint` now runs `eslint --max-warnings=0`. A warning becomes a failure. Starter philosophy: if it's worth warning about, it's worth fixing.
- Fixed the 1 error + 9 warnings the now-functional lint surfaced:
  - `src/app/(admin)/admin/2fa/page.tsx` — `Date.now()` in a Server Component (false positive for `react-hooks/purity`; added a one-line disable with reason). `<img>` for the QR code (data URL — `next/image` adds no value; one-line disable with reason).
  - `eslint.config.mjs` + `postcss.config.mjs` — assigned default exports to a `const` first.
  - `src/lib/rate-limit.ts` — removed six stale `eslint-disable` directives that no longer suppressed anything.

### Internal: CI action versions

- Bumped `actions/checkout@v4` → `@v5` and `actions/setup-node@v4` → `@v5`. GitHub announced Node 20 deprecation on Actions runners (forced to Node 24 June 2026, removed September 2026). v5 of both actions runs on Node 24.

## 0.3.0 — 2026-05-17

### Feature: Toast notifications (Sonner)

**Value:** Every action-triggering UI in the admin shell now has a consistent feedback channel. The starter ships with `<Toaster theme="system" richColors />` mounted in the root layout. Client components import `toast` directly from `sonner` and call `toast.success()` / `toast.error()` — no wrapper, no `next-themes`, no domain-specific abstraction. The `TwoFactorCard` from v0.2 (which surfaced action results via inline `useState` because there was no toast) now uses this pattern as the canonical example.

A new `ActionResult<T>` type in `src/types/actions.ts` formalizes the server-action return shape that every client component reads before calling `toast()`. Five unit tests pin the discriminated union's narrowing behavior.

### Feature: Self-serve account page

**Value:** Every signed-in user reaches `/account` to manage their own profile — the counterpart to the admin-only `/admin/users/[id]` shipped in v0.2. The page composes five cards:

- **Profile.** Edit display name. Audited.
- **Email.** Editable with email-link re-verification. The new email is held pending in a new `email_verification_tokens` table; only after the user clicks the link in their inbox is `users.email` updated. Google-only users can still change their display email — their OAuth provider account-id stays intact.
- **Password.** Change requires the current password; hidden entirely for Google-only users. Audited. Now also clears any in-flight password-reset tokens for the user.
- **2FA status.** Pill showing enrollment state + a link to `/account/2fa`.
- **Delete account.** Skeleton button + shadcn AlertDialog confirm + stub action that writes the audit row but does not yet cascade-delete. Intentional — every fork's data graph is different. A TODO in the action documents the path.

A new `/account/2fa` page exposes TOTP enrollment to every signed-in user (previously only admins via `/admin/2fa` could enroll). And a new unauthenticated landing at `/verify-email/[token]` lives in its own `(email-verify)` route group with no layout, so unauth click-from-email works.

Schema: new `email_verification_tokens` table with cascade FK on `userId`, unique index on `token`, unique index on `userId` (one pending change per user). Migration `0001_rapid_thor_girl.sql`.

### Feature: Forgot-password reset flow

**Value:** Completes the Credentials provider. Until v0.3 a user who forgot their password had no recovery path. The flow:

1. **`/signin`** has a new "Forgot password?" link in the Credentials block.
2. **`/forgot-password`** is the email-entry page. Submitting always shows the same "check your email" card regardless of whether the email exists, has a password, or is Google-only — full enumeration defense.
3. The email contains a link to **`/reset-password?token=<raw>`**.
4. Reset → `/signin` with a success toast.

Tokens are stored as SHA-256 hex at rest (a DB leak doesn't expose live reset URLs). 60-minute expiry. Race-safe: a unique index on `userId` enforces one in-flight reset per user; the action deletes any prior row before inserting.

Both `(password-reset)` pages and the two flow routes live in a new route group, mirrored on the `(email-verify)` pattern. Two new `PUBLIC_PATHS` entries (`/forgot-password`, `/reset-password`). Migration `0002_curious_famine.sql` adds `password_reset_tokens`.

### Feature: Rate limiting

**Value:** Five authentication-adjacent endpoints now reject abusive request rates with friendly errors and an audit trail. The starter ships with an in-memory limiter (lazy eviction, FIFO bound at 10,000 keys, no `setInterval`) that works zero-config on any single-instance deployment, plus an opt-in Upstash Redis backend that activates automatically when `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` are present in env.

Per-endpoint defaults:

| Endpoint | Limit | Key |
|---|---|---|
| `signIn` Credentials authorize | 5/min | `signin:<ip>:<email>` |
| `requestPasswordReset` | 5/hour | `pwreset_req:<ip>` |
| `consumeResetToken` | 10/hour | `pwreset_consume:<sha256(token)>` |
| `requestEmailChange` | 3/hour | `email_change:<userId>` |
| `verifyTotpAction` | 10/min | `totp:<userId>` |

Every block writes a `RATE_LIMIT_BLOCKED` audit row (new entry in `AUDIT_ACTIONS`). TOTP rejection travels via `?error=rate_limited` query param; the other four return `{ ok: false, error: ... }` per `ActionResult`. Limits fire *before* any DB read or `bcrypt.compare` on signin / password-reset paths — timing-side-channel defense.

**Escape hatch:** `RATE_LIMIT_DISABLED=true` in `.env.local` short-circuits `checkRateLimit` to `{ allowed: true }`. Required for running the e2e suite repeatedly against a long-lived dev server (the in-memory store accumulates state). Documented in `.env.example` with a strong "NEVER set in production" warning. Three regression tests pin the escape-hatch invariant (no audit leaks through it).
