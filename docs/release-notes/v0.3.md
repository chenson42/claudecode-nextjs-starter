## 0.3.1 — 2026-05-17

### Fix: lint was silently failing in CI

The `npm run lint` script invoked ESLint without a config file. ESLint 9 dropped support for `.eslintrc.*` as the default, so it errored out immediately. The CI workflow had `continue-on-error: true` on the lint step, which masked the failure as success — meaning the lint gate had been a no-op since the repo was created.

**What changed:**

- Added flat-config `eslint.config.mjs` that consumes `eslint-config-next`'s bundled flat-config array (covers next, react, react-hooks, typescript, jsx-a11y, import). Ignores `.next/`, `out/`, `build/`, `coverage/`, `test-results/`, `playwright-report/`, `drizzle/`, `next-env.d.ts`.
- Removed `continue-on-error: true` from the CI Lint step — failures now actually block the build.
- Strict-warning mode: `npm run lint` now runs `eslint --max-warnings=0`. A warning becomes a failure. Starter philosophy: if it's worth warning about, it's worth fixing.
- Fixed the 1 error + 9 warnings the now-functional lint surfaced:
  - `src/app/(admin)/admin/2fa/page.tsx` — `Date.now()` in a Server Component (false positive for `react-hooks/purity`; added a one-line disable with reason). `<img>` for the QR code (data URL — `next/image` adds no value; one-line disable with reason).
  - `eslint.config.mjs` + `postcss.config.mjs` — assigned default exports to a `const` first.
  - `src/lib/rate-limit.ts` — removed six stale `eslint-disable` directives that no longer suppressed anything.

### Internal: CI action versions

- Bumped `actions/checkout@v4` → `@v5` and `actions/setup-node@v4` → `@v5`. GitHub announced Node 20 deprecation on Actions runners (forced to Node 24 June 2026, removed September 2026). v5 of both actions runs on Node 24.

## 0.3.0 — 2026-05-17

### Feature: Toast notifications (Sonner)

**Value:** Every action-triggering UI in the admin shell now has a consistent feedback channel. The starter ships with `<Toaster theme="system" richColors />` mounted in the root layout. Client components import `toast` directly from `sonner` and call `toast.success()` / `toast.error()` — no wrapper, no `next-themes`, no domain-specific abstraction. The `TwoFactorCard` from v0.2 (which surfaced action results via inline `useState` because there was no toast) now uses this pattern as the canonical example.

A new `ActionResult<T>` type in `src/types/actions.ts` formalizes the server-action return shape that every client component reads before calling `toast()`. Five unit tests pin the discriminated union's narrowing behavior.

### Feature: Self-serve account page

**Value:** Every signed-in user reaches `/account` to manage their own profile — the counterpart to the admin-only `/admin/users/[id]` shipped in v0.2. The page composes five cards:

- **Profile.** Edit display name. Audited.
- **Email.** Editable with email-link re-verification. The new email is held pending in a new `email_verification_tokens` table; only after the user clicks the link in their inbox is `users.email` updated. Google-only users can still change their display email — their OAuth provider account-id stays intact.
- **Password.** Change requires the current password; hidden entirely for Google-only users. Audited. Now also clears any in-flight password-reset tokens for the user.
- **2FA status.** Pill showing enrollment state + a link to `/account/2fa`.
- **Delete account.** Skeleton button + shadcn AlertDialog confirm + stub action that writes the audit row but does not yet cascade-delete. Intentional — every fork's data graph is different. A TODO in the action documents the path.

A new `/account/2fa` page exposes TOTP enrollment to every signed-in user (previously only admins via `/admin/2fa` could enroll). And a new unauthenticated landing at `/verify-email/[token]` lives in its own `(email-verify)` route group with no layout, so unauth click-from-email works.

Schema: new `email_verification_tokens` table with cascade FK on `userId`, unique index on `token`, unique index on `userId` (one pending change per user). Migration `0001_rapid_thor_girl.sql`.

### Feature: Forgot-password reset flow

**Value:** Completes the Credentials provider. Until v0.3 a user who forgot their password had no recovery path. The flow:

1. **`/signin`** has a new "Forgot password?" link in the Credentials block.
2. **`/forgot-password`** is the email-entry page. Submitting always shows the same "check your email" card regardless of whether the email exists, has a password, or is Google-only — full enumeration defense.
3. The email contains a link to **`/reset-password?token=<raw>`**.
4. Reset → `/signin` with a success toast.

Tokens are stored as SHA-256 hex at rest (a DB leak doesn't expose live reset URLs). 60-minute expiry. Race-safe: a unique index on `userId` enforces one in-flight reset per user; the action deletes any prior row before inserting.

Both `(password-reset)` pages and the two flow routes live in a new route group, mirrored on the `(email-verify)` pattern. Two new `PUBLIC_PATHS` entries (`/forgot-password`, `/reset-password`). Migration `0002_curious_famine.sql` adds `password_reset_tokens`.

### Feature: Rate limiting

**Value:** Five authentication-adjacent endpoints now reject abusive request rates with friendly errors and an audit trail. The starter ships with an in-memory limiter (lazy eviction, FIFO bound at 10,000 keys, no `setInterval`) that works zero-config on any single-instance deployment, plus an opt-in Upstash Redis backend that activates automatically when `UPSTASH_REDIS_REST_URL` + `UPSTASH_REDIS_REST_TOKEN` are present in env.

Per-endpoint defaults:

| Endpoint | Limit | Key |
|---|---|---|
| `signIn` Credentials authorize | 5/min | `signin:<ip>:<email>` |
| `requestPasswordReset` | 5/hour | `pwreset_req:<ip>` |
| `consumeResetToken` | 10/hour | `pwreset_consume:<sha256(token)>` |
| `requestEmailChange` | 3/hour | `email_change:<userId>` |
| `verifyTotpAction` | 10/min | `totp:<userId>` |

Every block writes a `RATE_LIMIT_BLOCKED` audit row (new entry in `AUDIT_ACTIONS`). TOTP rejection travels via `?error=rate_limited` query param; the other four return `{ ok: false, error: ... }` per `ActionResult`. Limits fire *before* any DB read or `bcrypt.compare` on signin / password-reset paths — timing-side-channel defense.

**Escape hatch:** `RATE_LIMIT_DISABLED=true` in `.env.local` short-circuits `checkRateLimit` to `{ allowed: true }`. Required for running the e2e suite repeatedly against a long-lived dev server (the in-memory store accumulates state). Documented in `.env.example` with a strong "NEVER set in production" warning. Three regression tests pin the escape-hatch invariant (no audit leaks through it).

### Environment Variables

| Variable | Purpose | Required? |
|---|---|---|
| `NEXT_PUBLIC_APP_URL` | Origin used to build email links (verify-email + password-reset). | Yes (new in v0.3) |
| `UPSTASH_REDIS_REST_URL` | Upstash Redis REST endpoint. Activates the distributed rate limiter when present. | No |
| `UPSTASH_REDIS_REST_TOKEN` | Upstash auth token. Paired with the URL above. | No |
| `RATE_LIMIT_DISABLED` | Set to `true` in `.env.local` to short-circuit the limiter during e2e iteration. **Never set in production.** | No |
