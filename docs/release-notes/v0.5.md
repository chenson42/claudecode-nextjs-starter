## 0.5.0 ‚Äî 2026-05-18

### Feature: Commit-Message Standards + Escape-Rate Reporter

**Value:** The 7-day retrospective had no quantitative anchor. Every retro was qualitative observation only, which made it hard to tell a real trend ("we're escaping more bugs to production lately") from anecdote ("I noticed a couple of bad commits last week"). This release gives the retrospective a real number: a rolling 30-day escape rate, broken down by how each bug was eventually caught. The classification has to come from the commit author, so it's enforced at commit time by a git hook installed automatically on `npm install`.

The deeper bet: as the pipeline catches more bugs in-process (Phase 5 QA, periodic reviews, the new upstream-sync), the bugs that *do* commit get more interesting. Their distribution across catch channels tells you which gates are working and which aren't. That's the signal the retrospective has been missing.

#### What's New

- **Conventional Commits grammar enforced at commit time.** Subject line must match `<prefix>(<optional-scope>): <description>` with prefix from `feat | fix | chore | docs | test | refactor | style | perf | build | ci`. `Merge `, `Revert `, and `Release ` prefixes are exempt. Description is 1‚Äì100 characters.
- **`fix:` commits require two trailers:**
  - `Caught-By: automated-test | agent-review | human-review | production`
  - `Discovered-In: Phase-1 | Phase-2 | Phase-3 | Phase-4 | Phase-5 | Phase-6 | post-merge | production | retrospective`
- **The `Caught-By` boundary rule** lives in `CLAUDE.md`: if CI would have caught the bug without any agent judgment, use `automated-test`. If an agent had to decide to run a non-mandatory check, use `agent-review`. Two non-overlapping conditions, intentionally ‚Äî the metric only works if the classification is consistent.
- **`npm run stats:escape`** prints a 30-day rolling report: per-channel breakdown (absolute counts and percentages), total `fix:` count, missing-trailers count (the bypass-and-pre-cutoff indicator), and the grandfather cutoff date in the header. Output is stdout; tech-lead copies it into the weekly retrospective.
- **Mixed-commit rule:** one prefix per commit. A commit that adds a feature and fixes a bug must be split. The validator passively enforces this (only one prefix is accepted); `CLAUDE.md` explains why.
- **Hook installs automatically.** `scripts/install-hooks.sh` runs via the npm `prepare` lifecycle (no `husky` dependency), is idempotent, backs up any existing `commit-msg` hook, and is a no-op when `.git/` is absent (safe in deploy environments).
- **Grandfather cutoff: 2026-05-18.** Commits before this date lack the new trailers by design; `stats:escape` declares the cutoff in its header so the first retrospective's number is honest.
- **`pre-push` integration: skipped on purpose.** Forcing `stats:escape` at push time surfaces problems too late for the author to act on. Reconsider after three retrospectives if bypass rates persist.

#### Out of Scope / Deferred

- **MTTR per channel.** Would require a third `Fixes-Bug: <sha>` trailer and substantially more per-commit friction. Deferred to a future work-log.
- **In-pipeline catch rate.** Most bugs in this pipeline get caught and fixed *inside* Phase 5 and never become commits ‚Äî they don't show up in `git log` at all. `stats:escape` measures the escape rate only. A complementary tool that mines work-logs for in-pipeline catches is a separate feature.

#### Known Footguns

- **First retrospective.** Until `fix:` commits using the new trailers accumulate, `stats:escape` will show `0 fixes, all pre-cutoff` for ~weeks. That's correct, not broken. The header makes this self-explanatory.
- **Self-bypass via `Edit` + `git push` without `git commit`.** The hook only fires on commit. An agent that edits files and pushes via some non-commit path would bypass the standard. In a solo repo this is theoretical; in a team setup it might warrant a `pre-push` integration later.
